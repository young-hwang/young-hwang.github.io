# Smoke Test

> 이 글에서는 Kubernetes 클러스터가 올바르게 작동하는지 확인하기 위한 일련의 테스트를 수행합니다. Secret 암호화, Deployment, Port Forwarding, 로그, Exec, NodePort Service 등 핵심 기능들을 검증합니다.

---

## Smoke Test 체크리스트

| 테스트 항목 | 검증 내용 |
|-------------|-----------|
| **Data Encryption** | Secret이 etcd에 암호화되어 저장되는지 확인 |
| **Deployments** | Pod 생성 및 스케줄링이 정상 동작하는지 확인 |
| **Port Forwarding** | `kubectl port-forward` 명령이 동작하는지 확인 |
| **Logs** | `kubectl logs` 명령으로 컨테이너 로그 조회 가능한지 확인 |
| **Exec** | `kubectl exec` 명령으로 컨테이너 내 명령 실행 가능한지 확인 |
| **Services** | NodePort 서비스를 통한 외부 접근이 가능한지 확인 |

## 1. Data Encryption 테스트

Secret 데이터가 etcd에 암호화되어 저장되는지 확인합니다.

### Secret 생성

```bash
# Secret 생성
kubectl create secret generic kubernetes-the-hard-way \
  --from-literal="mykey=mydata"

# Secret 확인
kubectl get secret kubernetes-the-hard-way
kubectl get secret kubernetes-the-hard-way -o yaml
```

### Secret 데이터 확인

```bash
# base64 인코딩된 데이터 확인
kubectl get secret kubernetes-the-hard-way \
  -o jsonpath='{.data.mykey}' ; echo

# base64 디코딩하여 원본 데이터 확인
kubectl get secret kubernetes-the-hard-way \
  -o jsonpath='{.data.mykey}' | base64 -d ; echo
```

```
mydata
```

### etcd에서 암호화 확인

etcd에 직접 접근하여 데이터가 암호화되어 저장되었는지 확인합니다.

```bash
# etcd에서 Secret 데이터 직접 조회
ssh root@server \
  'etcdctl get /registry/secrets/default/kubernetes-the-hard-way | hexdump -C'
```

```
00000000  2f 72 65 67 69 73 74 72  79 2f 73 65 63 72 65 74  |/registry/secret|
00000010  73 2f 64 65 66 61 75 6c  74 2f 6b 75 62 65 72 6e  |s/default/kubern|
00000020  65 74 65 73 2d 74 68 65  2d 68 61 72 64 2d 77 61  |etes-the-hard-wa|
00000030  79 0a 6b 38 73 3a 65 6e  63 3a 61 65 73 63 62 63  |y.k8s:enc:aescbc|
00000040  3a 76 31 3a 6b 65 79 31  3a 44 61 dc 08 37 97 eb  |:v1:key1:Da..7..|
00000050  d4 d0 5b 14 39 23 a5 74  1b 3c a4 56 e4 a1 d1 17  |..[.9#.t.<.V....|
...
```

### hexdump 출력 해석

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                         etcd Secret 저장 형식                                │
│                                                                             │
│  /registry/secrets/default/kubernetes-the-hard-way  ← etcd key (평문)       │
│                                                                             │
│  k8s:enc:aescbc:v1:key1:<encrypted_data>            ← 암호화된 value        │
│  ───┬── ───┬─── ─┬─ ──┬─ ────────┬────────                                 │
│     │     │     │    │          │                                          │
│     │     │     │    │          └─ 암호화된 실제 데이터                      │
│     │     │     │    └─ 사용된 encryption key 이름                          │
│     │     │     └─ encryption provider 버전                                 │
│     │     └─ 암호화 알고리즘 (AES-CBC)                                      │
│     └─ Kubernetes 암호화 포맷 식별자                                        │
│                                                                             │
│  ✓ "mydata" 텍스트가 보이지 않음 = 암호화 성공!                              │
└─────────────────────────────────────────────────────────────────────────────┘
```

> **중요**: etcd key 경로(`/registry/secrets/...`)는 항상 평문이지만, value(실제 Secret 데이터)는 암호화되어 있습니다. hexdump에서 `mydata`가 보이지 않으면 암호화가 정상적으로 동작하는 것입니다.

## 2. Deployments 테스트

Pod 생성 및 스케줄링이 정상 동작하는지 확인합니다.

### Deployment 생성

```bash
# nginx Deployment 생성
kubectl create deployment nginx --image=nginx:latest

# 2개의 replica로 확장
kubectl scale deployment nginx --replicas=2

# Pod 상태 확인
kubectl get pod -owide
```

```
NAME                     READY   STATUS    RESTARTS   AGE    IP           NODE
nginx-54c98b4f84-pxp6c   1/1     Running   0          108s   10.200.1.2   node-1
nginx-54c98b4f84-qxpbn   1/1     Running   0          12s    10.200.0.2   node-0
```

### 확인 사항

- **STATUS**: `Running` 상태인지 확인
- **IP**: Pod IP가 각 노드의 Pod CIDR에서 할당되었는지 확인
  - node-0: 10.200.0.x
  - node-1: 10.200.1.x
- **NODE**: Pod가 두 노드에 분산 배치되었는지 확인

### 컨테이너 런타임 확인

```bash
# 각 노드에서 컨테이너 확인
ssh node-0 crictl ps
ssh node-1 crictl ps

# 프로세스 트리 확인
ssh node-0 pstree -ap
ssh node-1 pstree -ap
```

### 네트워크 구성 확인

```bash
# CNI 브리지 확인
ssh node-0 brctl show
ssh node-1 brctl show

# veth 인터페이스 확인 (Pod별 생성됨)
ssh node-0 ip addr
ssh node-1 ip addr
```

### Pod 네트워크 통신 테스트

Control Plane(server)에서 Pod IP로 직접 접근할 수 있는지 확인합니다.

```bash
# server에서 각 Pod로 HTTP 요청
ssh server curl -s 10.200.1.2 | grep title
ssh server curl -s 10.200.0.2 | grep title
```

```html
<title>Welcome to nginx!</title>
```

> 이 테스트가 성공하면 이전 단계에서 설정한 Pod 네트워크 라우팅이 정상 동작하는 것입니다.

## 3. Port Forwarding 테스트

`kubectl port-forward` 명령이 동작하는지 확인합니다.

### Port Forward 설정

```bash
# Pod 이름 가져오기
POD_NAME=$(kubectl get pods -l app=nginx -o jsonpath="{.items[0].metadata.name}")
echo $POD_NAME

# 로컬 8080 포트를 Pod의 80 포트로 포워딩 (백그라운드 실행)
kubectl port-forward $POD_NAME 8080:80 &

# 프로세스 확인
ps -ef | grep kubectl
```

### Port Forward 테스트

```bash
# HTTP 요청 테스트
curl --head http://127.0.0.1:8080
```

```
HTTP/1.1 200 OK
Server: nginx/1.27.0
Date: Sat, 04 Jan 2025 07:30:00 GMT
Content-Type: text/html
Content-Length: 615
...
```

### Port Forward 동작 원리

```

│                                                                             │

│  │ curl            │      │ kubectl         │      │ API Server      │     │

│                                                    │ (nginx)         │     │
```

## 4. Logs 테스트

`kubectl logs` 명령으로 컨테이너 로그를 조회할 수 있는지 확인합니다.

### 로그 조회

```bash
# Pod 로그 확인
kubectl logs $POD_NAME
```

### 로그 생성 및 확인

```bash
# HTTP 요청으로 로그 생성
curl --head http://127.0.0.1:8080

# 로그에 요청이 기록되었는지 확인
kubectl logs $POD_NAME
```

```
127.0.0.1 - - [04/Jan/2025:07:30:00 +0000] "HEAD / HTTP/1.1" 200 0 "-" "curl/7.88.1" "-"
```

### Port Forward 종료

```bash
# kubectl port-forward 프로세스 종료
kill -9 $(pgrep kubectl)
```

## 5. Exec 테스트

`kubectl exec` 명령으로 컨테이너 내에서 명령을 실행할 수 있는지 확인합니다.

### 컨테이너 내 명령 실행

```bash
# nginx 버전 확인
kubectl exec -ti $POD_NAME -- nginx -v
```

```
nginx version: nginx/1.27.0
```

### 다양한 Exec 예시

```bash
# 컨테이너 쉘 접속
kubectl exec -ti $POD_NAME -- /bin/bash

# 파일 시스템 확인
kubectl exec $POD_NAME -- ls -la /usr/share/nginx/html

# 환경 변수 확인
kubectl exec $POD_NAME -- env
```

| 옵션 | 설명 |
|------|------|
| `-t` | TTY 할당 (터미널 출력 포맷팅) |
| `-i` | 표준 입력 유지 (대화형 모드) |
| `--` | kubectl 옵션과 컨테이너 명령 구분자 |

## 6. Service (NodePort) 테스트

NodePort 서비스를 통해 외부에서 Pod에 접근할 수 있는지 확인합니다.

### NodePort Service 생성

```bash
# nginx Deployment를 NodePort 서비스로 노출
kubectl expose deployment nginx --port 80 --type NodePort

# Service와 Endpoints 확인
kubectl get service,ep nginx
```

```
NAME            TYPE       CLUSTER-IP    EXTERNAL-IP   PORT(S)        AGE
service/nginx   NodePort   10.32.0.149   <none>        80:31410/TCP   10s

NAME              ENDPOINTS                     AGE
endpoints/nginx   10.200.0.2:80,10.200.1.2:80   10s
```

### 출력 해석

| 항목 | 값 | 설명 |
|------|-----|------|
| TYPE | NodePort | 서비스 유형 |
| CLUSTER-IP | 10.32.0.149 | 클러스터 내부 IP |
| PORT(S) | 80:31410/TCP | 서비스포트:노드포트 |
| ENDPOINTS | 10.200.0.2:80, 10.200.1.2:80 | 실제 Pod IP:포트 |

### NodePort Service 동작 원리

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                        NodePort Service 동작                                 │
│                                                                             │
│  외부 클라이언트                                                             │
│        │                                                                    │
│        ▼                                                                    │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │              node-0:31410 또는 node-1:31410                          │   │
│  │                        (NodePort)                                    │   │
│  └──────────────────────────────┬──────────────────────────────────────┘   │
│                                 │                                          │
│                                 ▼                                          │
│  ┌─────────────────────────────────────────────────────────────────────┐   │
│  │                    iptables (kube-proxy 규칙)                        │   │
│  │                                                                      │   │
│  │   DNAT: 31410 → 10.32.0.149:80 (ClusterIP)                          │   │
│  │   Load Balance: 10.200.0.2:80 또는 10.200.1.2:80                    │   │
│  └──────────────────────────────┬──────────────────────────────────────┘   │
│                                 │                                          │
│              ┌──────────────────┼──────────────────┐                       │
│              ▼                                     ▼                       │
│  ┌───────────────────┐                 ┌───────────────────┐              │
│  │ Pod (10.200.0.2)  │                 │ Pod (10.200.1.2)  │              │
│  │ node-0            │                 │ node-1            │              │
│  └───────────────────┘                 └───────────────────┘              │
└─────────────────────────────────────────────────────────────────────────────┘
```

### NodePort 접근 테스트

```bash
# NodePort 번호 가져오기
NODE_PORT=$(kubectl get svc nginx --output=jsonpath='{range .spec.ports[0]}{.nodePort}')
echo $NODE_PORT

# 각 노드를 통해 접근 테스트
curl -s -I http://node-0:${NODE_PORT}
curl -s -I http://node-1:${NODE_PORT}
```

```
HTTP/1.1 200 OK
Server: nginx/1.27.0
...
```

> **모든 노드에서 접근 가능**: NodePort는 클러스터의 모든 노드에서 동일한 포트로 서비스에 접근할 수 있습니다. 실제 Pod가 해당 노드에 없어도 kube-proxy가 트래픽을 올바른 Pod로 라우팅합니다.

## 트러블슈팅

### Pod가 Pending 상태

```bash
kubectl describe pod <pod-name>
```

**확인 사항**:
- 노드가 Ready 상태인지: `kubectl get nodes`
- 리소스 부족인지: Events 섹션 확인

### Port Forward 실패

```
error: unable to forward port because pod is not running
```

**해결**:
```bash
# Pod 상태 확인
kubectl get pod $POD_NAME

# Pod가 Running이 아니면 재생성
kubectl delete pod $POD_NAME
```

### Exec 실패

```
error: unable to upgrade connection: pod does not exist
```

**확인 사항**:
- Pod 이름이 정확한지 확인
- kubelet 상태 확인: `ssh node-0 systemctl status kubelet`

### NodePort 접근 실패

```bash
# iptables 규칙 확인
ssh node-0 iptables -t nat -L KUBE-SERVICES -n | grep <nodeport>

# kube-proxy 상태 확인
ssh node-0 systemctl status kube-proxy
```

## 마무리

이번 단계에서 완료한 작업:

- **Data Encryption**: Secret이 etcd에 AES-CBC로 암호화되어 저장됨 확인
- **Deployments**: Pod 생성, 스케줄링, 노드 분산 배치 확인
- **Port Forwarding**: 로컬 포트에서 Pod로 트래픽 전달 확인

- **Exec**: 컨테이너 내 명령 실행 기능 확인
- **Services**: NodePort를 통한 외부 접근 확인

모든 Smoke Test가 통과하면 Kubernetes The Hard Way 클러스터가 정상적으로 동작하는 것입니다.

**Kubernetes The Hard Way** 실습을 완료했습니다. 이 과정을 통해 다음을 직접 구축하고 이해했습니다:

- TLS 인증서 및 PKI 구성
- kubeconfig 파일 구조
- etcd 분산 저장소
- Control Plane 컴포넌트 (API Server, Controller Manager, Scheduler)
- Worker Node 컴포넌트 (kubelet, kube-proxy, containerd)
- CNI 네트워크 플러그인
- Pod 네트워크 라우팅
- RBAC 권한 관리
- Secret 암호화
